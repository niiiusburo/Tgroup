Face ID Complete Loop

Option C means V1 ships enrollment, verification, search, and failure handling together.

Operational Flow
One engine, two staff workflows
1. Lock Face To Customer
Open customer profile or edit modal
↓
Capture face with camera overlay
↓
Local service detects face and returns embedding
↓
API stores embedding linked to partner id
↓
Customer shows Face ID registered state
2. Search Customer By Face
Click Face ID search on /customers
↓
Capture walk-in face
↓
API compares embedding against registered customers
↓
High-confidence match opens customer profile
↓
No-match keeps image available for new customer registration
Decision States

Good Capture

Exactly one face, quality passes, embedding stored or searched.

No Face / Many Faces

Do not store. Ask staff to retake with one visible customer face.

Low Confidence

Do not auto-open a record. Show closest candidates only if we explicitly allow staff review.

Duplicate Risk

When a new face looks close to an existing customer, require confirmation before overwriting or creating a second enrollment.

Design Check

For V1, should low-confidence search show staff possible matches, or should it only say no match?

A

Strict: No Match Only

Safer and simpler. The app only opens a customer when confidence is high enough.

B

Staff Candidate Review

More useful for imperfect captures. Show the top 3 possible customers, but require staff to choose manually.
