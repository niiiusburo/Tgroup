# ADR-0001: Root Authority Stack

## Status
Accepted

## Context
TGClinic had strong local docs in `AGENTS.md`, `website/agents.md`, `website/design.md`, `product-map/`, and `notes/`, but it did not have a single root authority-doc routing pattern. Agents and humans were reading different files depending on who wrote the session prompt, leading to inconsistent decisions and duplicate policy.

## Decision
Create and use the root authority stack:
1. `AGENTS.md` — workflow, ownership, coordination, reporting, escalation.
2. `ARCHITECTURE.md` — repo structure, dependency direction, domain boundaries.
3. `DESIGN.md` — visual system routing and design constraints.
4. `BEHAVIOR.md` — interaction behavior, forms, errors, loading, permissions.
5. `DECISIONS.md` — accepted project decisions and historical rationale.
6. `COORDINATION_REQUESTS.md` — cross-domain blockers.
7. `IDEA.md` — non-authority inbox.
8. `docs/runbooks/` — operational authority.

If an ad hoc prompt conflicts with the authority stack, follow the authority stack unless the user explicitly says they are changing policy.

## Consequences
- Future work starts from the root authority stack, then drills into `product-map/`, `website/`, and `notes/` as needed.
- `notes/` remains supporting context, not the highest source of truth.
- Every agent session must read the relevant authority files before editing.

## Alternatives Rejected
- **Single monolithic doc:** Would become too large and discourage updates. Splitting by concern keeps files manageable.
- **No stack, rely on `notes/`:** `notes/` was historical reference, not policy. Operational drift occurred because agents treated notes as rules.
