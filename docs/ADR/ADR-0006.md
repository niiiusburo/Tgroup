# ADR-0006: Frontend Location Filter Only (Backend Does Not Enforce Location Scope on Lists)

## Status
Accepted

## Context
Multi-location clinics need staff to see only their assigned branches. The backend historically lacked consistent location scoping on list routes.

## Decision
The frontend `LocationContext` is the primary location filter. Backend list routes generally return all records and rely on the frontend to filter by `companyId`. Exception: specific routes that explicitly implement location scoping (documented per route).

## Consequences
- A broken or bypassed frontend could expose records from other locations.
- Simpler backend queries; no repeated `companyid = $1` boilerplate in every list handler.
- If a route needs backend enforcement, it must be explicitly documented in the route handler and the domain YAML.

## Alternatives Rejected
- **Backend-enforced location scope on every route:** Would require touching 30+ route files and risk performance degradation. Planned for a future v2 API cleanup.
- **Row-level security (RLS) in PostgreSQL:** Too complex for the current shared-pool architecture and would conflict with admin users who need cross-location views.
