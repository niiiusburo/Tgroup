1

What permission should protect the /services page?

Right now the Services page is gated by customers.edit, which makes no sense — it’s not about customers.

Think of it like keys to rooms:

Pick this
🟢 services.view
A "Services" key opens the Services room. Clean and obvious.
  • Staff with services.view can open the page
  • Staff with services.edit can change data
Don't pick
🔴 customers.edit
Using a "Customer" key to open the "Services" room is confusing and hides real access.
  • A receptionist who can edit customers might not need services
  • Creates hidden security holes
💡
Recommendation: Use services.view for read access and services.edit for mutations. This makes the key match the room.
2

What permission should control SaleOrders and treatment writes?

When a staff member creates a treatment plan or a sale order, which key should unlock that action?

Caution
🟡 customers.edit
If a treatment is part of a customer record, the Customer key can work — but only if treatments are truly customer edits.
Pick this
🟢 saleorders.edit or keep customers.edit
If treatments are customer-record edits, keep customers.edit. If they are independent sales actions, create saleorders.edit.

Recommendation: Keep customers.edit only if treatments are truly edits to the customer record. Otherwise, introduce saleorders.edit so sales actions have their own lock.

Should payment patch and payment proof use the same permission?

Patching a payment record is editing data. Uploading proof is adding a file. Same page, different actions.

Pick this
🟢 payment.add (proof) + payment.edit (patch)
Proof upload = adding something new (add). Record patch = changing existing data (edit).
Don't pick
🔴 payment.add for everything
If anyone who can upload a receipt can also edit the amount, that’s dangerous.

💡
Recommendation: Proof upload uses payment.add; record patch uses payment.edit. Registry and docs must match.
5

Should the Permission Board show only real, registry-backed permissions?

Right now the matrix generates permission strings from labels, so it shows fake or missing permissions that don’t exist in the real registry.

Pick this
🟢 Show only real permissions
The board reads from the canonical registry. If a permission isn’t in the registry, it doesn’t appear.
  • No fake buttons
  • No missing permissions
  • What you see = what the backend enforces
Don't pick
🔴 Generate from labels
Labels can be out of sync, so the board shows permissions that don’t exist or hides ones that do.
💡
Recommendation: Yes — the Permission Board must be a direct mirror of the registry. If it’s not in the code, it shouldn’t be on the board.
6

If an employee has no location scope set, what does that mean?

Empty location scope is ambiguous: their home branch? All branches? Nowhere?

Pick this
🟢 Primary branch only
If you don’t say where someone works, they work at their home branch. Safe default.
Risky
🟡 All branches
Empty means everywhere. A typo could give someone global access by accident.
Don't pick
🔴 No access
Empty means locked out. A new hire with no scope set can’t log in — support nightmare.
Location scope = empty
→
Home / primary branch
(safe default)
💡
Recommendation: Empty scope means primary branch only. It’s the least surprising behavior: if you don’t specify extra locations, you only see your own.
7

Should the system stop an admin from deleting their own permissions.edit?

An admin with permissions.edit can edit anyone’s permissions — including their own. Should they be allowed to remove it from themselves?

Pick this
🟢 Block or confirm
If you try to remove your own admin key, the system says "Are you sure?" or simply blocks it.
  • Prevents accidental self-lockout
  • Requires explicit confirmation if allowed at all
Don't pick
🔴 Allow silently
One wrong click and the only admin is locked out of permissions forever. Someone else must rescue them.
💡
Recommendation: Yes — block or require strong confirmation before an admin can remove their own permissions.edit. Think of it like a safety cover on a master key.
8

Should “Remember Me” actually last 60 days?

The login screen and docs tell users "Remember me for 60 days." But the API signs tokens that expire in 24 hours.

Pick this
🟢 Make it 60 days
If the checkbox says 60 days, the cookie/token should live for 60 days. Promise = reality.
Don't pick
🔴 Keep mismatch
Users think they stay logged in, but they’re kicked out tomorrow. Support tickets and frustration.
User checks "Remember me"
→
UI says 60 days
→
API must match: 60 days
💡
Recommendation: Yes — implement the 60-day token lifetime so the backend keeps the promise the frontend already makes.
