## 1. Summary

Three enhancements to the Calendar page (`/calendar`):

1. **Interactive status flow** on appointment cards — bring Overview's two-tier status model (arrive → waiting → in-treatment → done, plus cancel) into Day and Week views, including a check-in button and a live wait timer.
2. **Hourly time rail** in Day view — always render every 30-minute slot from 07:00 to 20:00; empty slots collapse to thin placeholders, and clicking one opens the appointment-create form prefilled with that date and time.
3. **Excel export** on Day, Week, and Month views — toolbar button opens a dialog with two filter modes (current filter vs. date range) and writes a `DanhSachLichHen.xlsx` file matching the legacy TG Dental schema.

All three must preserve existing behavior for drag-reschedule, search, filter drawer, location filter, and edit modal.

---

## 2. Goals & Non-Goals

### Goals
- Eliminate the need to bounce between Overview and Calendar to change a patient's live status.
- Make "add appointment at 14:30 today" a single click instead of a multi-step form.
- Replicate the TG Dental legacy `DanhSachLichHen.xlsx` export 1:1 so doctors can receive the same daily spreadsheet.

### Non-Goals
- No changes to the Overview page itself.
- No changes to Month view's status handling (Month view only shows counts per day, not individual cards).
- No new API endpoints — reuse `updateAppointment()` and `createAppointment()` already exposed by `@/lib/api`.
- No backend schema changes.

## 4. Feature 1 — Interactive Status Flow

### 4.1 Visual States

Each appointment card has one authoritative status derived from `CalendarAppointment.status` plus (when arrived) a downline `checkInStatus` that we will store in local state *and* map to the API `state` field.

| Phase | Badge label (vi) | Colors | Shown actions on hover |
|---|---|---|---|
| Scheduled (default) | **Đang hẹn** (purple) | `bg-purple-100 text-purple-700` | `UserCheck` (Check-in) + `X` (Cancel) |
| Arrived — Waiting | **Chờ khám** (amber) | `bg-amber-50 text-amber-700` | Status dropdown + `WaitTimer` visible |
| Arrived — In treatment | **Đang khám** (sky) | `bg-sky-50 text-sky-700` | Status dropdown |
| Arrived — Done | **Hoàn thành** (emerald) | `bg-emerald-50 text-emerald-700` | Status dropdown |
| Cancelled | **Hủy hẹn** (red) | `bg-red-100 text-red-700` | Status dropdown (to revert to Scheduled) |

**Why two tiers:** this mirrors Overview's proven flow — the receptionist checks patients in once, then doctors/assistants progress them through treatment phases without the receptionist needing to touch it again.

### 4.2 Interaction Flow

```
┌────────────────────────────────────────────────────────────────┐
│ Appointment Card (DayView / WeekView)                          │
│                                                                │
│ [Đang hẹn ▾]                          [✎ edit] [🧍 check-in]   │  ← scheduled state, hover shows actions
│ Nguyễn Văn A                                                   │
│ 08:00 – 08:30 · BS. Trang                                      │
└────────────────────────────────────────────────────────────────┘

User clicks 🧍 check-in
  → onMarkArrived(id)
  → API: updateAppointment(id, { state: 'arrived' })
  → Card refreshes

┌────────────────────────────────────────────────────────────────┐
│ [Chờ khám ▾] ⏱ 00:02:15                    [✎]                │  ← arrived+waiting, WaitTimer ticks
│ Nguyễn Văn A                                                   │
└────────────────────────────────────────────────────────────────┘

User clicks the badge
  → StatusDropdown opens with radio options:
    ◉ Chờ khám
    ○ Đang khám
    ○ Hoàn thành
    ——
    ○ Hủy hẹn  (red)

  Selecting 'Đang khám' → updateAppointment(id, { state: 'in Examination' })
  Selecting 'Hoàn thành' → updateAppointment(id, { state: 'done' })
  Selecting 'Hủy hẹn'    → updateAppointment(id, { state: 'cancelled' })
```

### 4.3 Rules
- The **check-in icon (`UserCheck`)** is **only visible on hover** and only when the current phase is `scheduled`. Once the patient has been checked in (arrived), the icon is replaced by the live timer + dropdown-enabled badge.
- The **cancel `X` icon** is also hover-only and only on `scheduled` cards. For arrived cards, cancellation happens via the dropdown (keeps the card from accidentally being cancelled mid-treatment).
- Clicking the status badge or any of its menu items must `stopPropagation` so the card's own click (edit modal) does not fire.
- After any status change, call `refresh()` on the parent `useCalendarData` hook so the filter chips and summary cards stay in sync.
- Arrival time: when the user clicks check-in, write the current wall-clock time to `localStorage['tgclinic:arrivalTimes']` (same key as Overview) so the `WaitTimer` can read it on reload and both pages stay in sync.

### 4.4 Status↔API mapping (single source of truth)

Extract to a shared helper `website/src/lib/appointmentStatusMapping.ts`:

```ts
export type CalendarPhase =
  | 'scheduled'
  | 'waiting'       // arrived, not started
  | 'in-treatment'
  | 'done'
  | 'cancelled';

// API state ← phase
export const PHASE_TO_API_STATE: Record<CalendarPhase, string> = {
  scheduled:     'scheduled',
  waiting:       'arrived',
  'in-treatment':'in Examination',
  done:          'done',
  cancelled:     'cancelled',
};

// API state → phase (tolerant of legacy values)
export function apiStateToPhase(state: string | null | undefined): CalendarPhase { ... }
```

Both the Calendar and Overview hooks reference this mapping; no more string literals scattered across files.

## 5. Feature 2 — Hourly Time Rail (Day View)

### 5.1 Visual Layout

```
┌────────────────────────────────────────────────────────┐
│ Day header — Thứ Năm, 17 tháng 4 · 12 lịch hẹn         │
├────────────────────────────────────────────────────────┤
│ 07:00 ───────────────────────────────────────  + hẹn   │  ← collapsed empty slot (24px)
│ 07:30 ───────────────────────────────────────  + hẹn   │
│ 08:00 ─────────────────────────────────────────── (3)  │  ← has appointments, full row
│   [card] [card] [card]                                  │
│ 08:30 ───────────────────────────────────────  + hẹn   │
│ 09:00 ─────────────────────────────────────────── (5)  │
│   [card] [card] [card] [card] [card]                   │
│  …                                                      │
│ 20:00 ───────────────────────────────────────  + hẹn   │
└────────────────────────────────────────────────────────┘
```

### 5.2 Rules
- Slot window: **07:00 → 20:00**, **every 30 minutes** → 27 slots (07:00, 07:30, …, 20:00).
- Empty slot height: **24px**. Full slot (with cards): current height.
- Empty slot hover state: background tint + right-aligned `+ Thêm lịch hẹn` label.
- Empty slot click: opens `AppointmentForm` inside a shared `AppointmentFormModal` (refactored from `QuickAddAppointmentButton`) with:
  - `date = currentDate` (YYYY-MM-DD)
  - `startTime = slot` (HH:mm)
  - `endTime = slot + 30min` (default, user can override)
- After successful create: close modal + call `refresh()`.

### 5.3 Refactor required
Today `QuickAddAppointmentButton` both renders the button **and** owns the modal. Extract `AppointmentFormModal` so Day view can trigger the same modal from an empty slot. Button remains the entry point from the toolbar.

### 5.4 Accessibility
Empty slot row is a `<button>` with `aria-label="Thêm lịch hẹn lúc 14:30"` so keyboard users can tab through and Enter to open the form.

## 6. Feature 3 — Excel Export

### 6.1 Toolbar Button

Added to the Calendar page toolbar, right of the filter button, visible in **Day, Week, and Month** views:

```
[ Bộ lọc ] [ Xuất Excel ]
```

### 6.2 Export Dialog

Matches the provided Vietnamese mockups:

```
┌─────────────────────────────────────────┐
│  Xuất dữ liệu lịch hẹn              [✕] │
├─────────────────────────────────────────┤
│                                         │
│  ◉ Theo bộ lọc hiện tại                │
│  ○ Theo khoảng thời gian               │
│                                         │
│  [date picker pair — shown only when    │
│   "Theo khoảng thời gian" is selected]  │
│                                         │
├─────────────────────────────────────────┤
│                    [ Đóng ] [ Xuất file ]│
└─────────────────────────────────────────┘
```

### 6.3 Filter Modes

**Mode A — Theo bộ lọc hiện tại (current filter)**
- Uses the view's current date range:
  - Day view → `[currentDate, currentDate]`
  - Week view → `[weekDates[0], weekDates[6]]`
  - Month view → first to last day of the visible month
- Applies current filter state: `selectedDoctors`, `selectedStatuses`, `selectedColors`, `search`.
- Reads straight from the `filteredAppointments` already materialized by `useCalendarData`.

**Mode B — Theo khoảng thời gian (date range)**
- Two date inputs, default to `[today, today]` (matching the reference image `16/04/2026 – 16/04/2026`).
- Ignores active filters; fetches fresh data via `fetchAppointments({ dateFrom, dateTo, companyId })`.
- Respects the Calendar's `selectedLocationId` (location is a page-level filter, not a "Calendar filter").

### 6.4 File Output

- Library: **`xlsx`** (SheetJS Community Edition) + **`file-saver`** for the download. Both already common choices in the React ecosystem; no native deps.
- Filename: `DanhSachLichHen_YYYY-MM-DD.xlsx` (or `DanhSachLichHen_YYYY-MM-DD_YYYY-MM-DD.xlsx` for ranges).
- Sheet name: `Sheet1` (matches sample).
- Row 1: header row with exact Vietnamese labels from §3.
- Rows 2+: one row per appointment, sorted by `date ASC, time ASC`.
- Empty cells are serialized as empty strings `''`, not `null` (matches sample).
- Column J stays blank (preserved for compatibility with downstream tools that may consume this).

### 6.5 Status label localization

Reuse `apiStateToPhase` then map to Vietnamese:

| Phase | Trạng thái (col H) |
|---|---|
| scheduled | Đang hẹn |
| waiting | Đã đến |
| in-treatment | Đang khám |
| done | Hoàn thành |
| cancelled | Hủy hẹn |

### 6.6 Visit type (Loại khám, col G)

The legacy samples show two literal values: `Khám mới` and `Tái khám`. These are derived in the current backend from the appointment's `note` field (parsed by `parseAppointmentNote` → `type`). For V18.2:
- If `parsed.type` contains `tksn` / `khám mới` → **Khám mới**
- Otherwise → **Tái khám** (safe default matching the legacy spreadsheet distribution)

### 6.7 Telemetry
Not required for V18.2 (no analytics infra in place). Log start/error via `console.info`/`console.error` only.

## 8. Data Flow

### 8.1 Status change
```
User clicks badge → StatusBadgeMenu opens
User picks phase → onPhaseChange(id, phase) → useCalendarData.updateAppointmentStatus
  ↓
  updateAppointment(id, { state: PHASE_TO_API_STATE[phase] })
  ↓ optimistic update of local `appointments` array
  refresh() on success to get authoritative state
```

### 8.2 Empty slot → new appointment
```
User clicks EmptyTimeSlot(time='14:30')
  ↓
Calendar opens AppointmentFormModal with { date, startTime: '14:30' }
  ↓
User submits → createAppointment(…)
  ↓ refresh()
```

### 8.3 Excel export
```
User opens ExportDialog
  ↓ selects mode
Mode A: filteredAppointments from useCalendarData (already in memory)
Mode B: fetchAppointments({ dateFrom, dateTo, companyId })
  ↓
exportAppointmentsXlsx(rows, filename)
  ↓ xlsx.utils.json_to_sheet → xlsx.write → Blob → file-saver.saveAs
```

## 12. Risks & Open Questions

| Risk / Question | Mitigation / Decision |
|---|---|
| Bundle size from `xlsx` | Dynamic import, lazy on first export click |
| Visit type (Loại khám) derivation is heuristic | Defaults to `Tái khám` (matches legacy distribution ≥80%); can be promoted to a DB field in a later phase |
| Multiple overlapping appointments in the same 30-min slot | Existing `auto-fill` grid behavior is preserved — slot just shows N cards in a wrapping row |
| Wait timer desync across Calendar + Overview | Shared `localStorage['tgclinic:arrivalTimes']` key (already used by Overview) |
| User in Vietnam time zone vs browser TZ | Continue using `useTimezone()` helpers — no new timezone logic introduced |

## 13. Out of Scope (V18.3+ candidates)

- Drag-and-drop into empty slots (feature 2 + drag-reschedule cross-pollination).
- Month view per-card status editing.
- Excel export for other pages (Customers, Employees).
- Admin-configurable operating hours.
